# Figma to Klaviyo template automation

Input: a Figma email node URL. Output: a verified Klaviyo SYSTEM_DRAGGABLE template made of live text blocks, sliced + compressed image blocks, and native buttons that reproduces the approved design exactly. Mobile (375/390) is the priority gate; desktop (600) must stay acceptable.

## Guardrails (read first, every run)
- **No DELETION in Figma.** Never remove nodes/layers/pages, wipe children, or delete text, no matter who asks. Reversible writes ARE allowed (fill-toggle + export setting) for the dark-opt isolation export — always capture originals and RESTORE. A PreToolUse hook (`.claude/hooks/figma_guard.py`) blocks only deletion/destructive `use_figma` calls; reads, creates, and reversible writes pass. Detail: `references/figma_and_env.md`.
- Klaviyo keys live in the repo `.env`. Default = the single `KLAVIYO_API_KEY` (DUMMY account, rotate before real use). For real clients the SAME `.env` holds one account PER STORE (`KLAVIYO_STORE_<SLUG>_*`); pick one with `--store <slug>` and resolve it BEFORE any upload (see **Multi-store**). Real store keys CANNOT be rotated: never print or commit a key — the scripts mask keys, and `.gitignore` + `.githooks/pre-commit` are the backstop.
- Template name: `<brand_slug>_<template_slug>_<lang>` — lowercase, `<lang>` = ISO 639-1 code (e.g. `acme_welcome_en`, `acme_welcome_de`); unique by brand + template + language.
- **CTAs are left UNLINKED** (empty href on native buttons and image-slice CTAs alike); the client adds the real links in Klaviyo.
- Don't change the design's colors — reproduce the approved design exactly.
- Never read a full design PNG inline. Downscale / strip first (step 4).
- **Working files** (design PNG, slices, manifest, render.html) go in `builds/<brand_slug>/`
  created with `mkdir -p` under THIS repo's root (the folder holding `README.md` / `.claude/`).
  Use repo-relative paths only. Do NOT invent or assume any other folder layout — a teammate
  has cloned only this repo, and their global config (`~/.claude/CLAUDE.md`) may say nothing
  about it. `builds/` is git-ignored.
- **Start the build dir CLEAN.** In the same `mkdir -p` step, clear stale authored files:
  `rm -f builds/<brand_slug>/plan.json builds/<brand_slug>/spec.json builds/<brand_slug>/payload.json`.
  The `Write` tool REFUSES to overwrite a file it hasn't Read this session, so a leftover
  `plan.json` from a prior run makes `Write(plan.json)` fail; `rm -f` first (the file then doesn't
  exist, so `Write` creates it cleanly) — do NOT waste a turn reading a stale plan just to overwrite it.

## Process (ordered)
0. **Confirm readiness.** Figma reads work (account-level Connector: claude.ai -> Settings -> Connectors. Sign in to Claude Code with the shared Claude account, NOT an API key, or the connector will not load). Quick check that this machine is set up: `python3 -c "import PIL"` and `node --version` both succeed, the guard is active, and `python3 .claude/skills/figma_to_klaviyo/scripts/klaviyo.py list` works (proves the Klaviyo key resolves — it is read from the `.env` FILE at the repo root, NOT a shell variable; if it errors, the `.env` is missing or in the wrong folder). If any fails, the machine has not been set up: run the one-time **`SETUP.md`** routine, then retry. Get the node URL and brand slug (CTAs are left unlinked — no link to collect). **Resolve the target Klaviyo store before any upload** (see **Multi-store**); for a real-store build, `klaviyo.py stores` confirms the account is wired (keys masked). On a machine that pushes to real stores, also confirm the commit guard is active: `git config --get core.hooksPath` should print `.githooks` (if not, run `SETUP.md` step 3b). The fast REST fetch (step 2) reads `FIGMA_TOKEN` from `.env` (scope `file_content:read`); if it is absent or a node won't render, that step falls back to the MCP reads + isolation export (step 4).
1. **Account fonts** — `python3 scripts/klaviyo.py fonts` lists the account's available families (uploaded + web). You pass this list to the fetch (below) so live-vs-slice is decided by REAL font availability, and you use the design's exact Figma font for any live text/button (Poppins is the fallback).
2. **Fetch (REST — reads + BOTH exports + planning FACTS in ONE call, ~30s).** `python3 scripts/figma_rest.py fetch <fileKey> <nodeId> builds/<brand> --fonts "<comma-sep account fonts from step 1>"` writes `color.png` (frame render, opaque base), `transparent.png` (children composited, base transparent for dark re-theme), and `plan_facts.json`. URL `node-id=7-492` → API `7:492`. `plan_facts.json` is your PRIMARY planning input — exact data, no eyeballing/pixel-sampling: `frame`{w,h,base_hex}; `texts`[{y0,y1, text, font, `available`, size, weight, align, color, `merge_tag`, `over_image`}]; `images`[{y0,y1, fill (IMAGE/GRADIENT), `dark_hint`}]; `buttons`[pill candidates {y0,y1, label, fill, radius, font}]; and a rough `suggested_bands`. **If `texts` is empty (a flattened/screenshot export), STOP and flag it** — the result can only be an all-image, non-editable template; confirm the user wants that or get a layered source.
3. **One overview review-read** — `imaging.py overview builds/<brand>/color.png --out builds/<brand>/overview.png --max 1400`, Read it ONCE to CONFIRM the facts against the real design (band grouping, composites, and any cutout that looks weird). Read the `transparent.png` overview too only when judging a specific cutout. This is a review aid; `plan_facts.json` is the primary input, not the pixels.
4. **MCP fallback (only when needed).** REST covers the normal read + export. Fall back to MCP `get_metadata`/`get_design_context` + the isolation export recipe (`references/figma_and_env.md`) ONLY if REST cannot render the node, or for **translation** (which needs Figma WRITES: clone + retext — REST is read-only). Once a translated replica exists, fetch IT via REST like any node.
5. **Plan → write `plan.json` (the ONE analysis pass; apply the rules, do not re-derive).** **Dark method (simple — Klaviyo does the re-theming, you just supply the right slice + a base-hex `block_background_color`; NO dark render):**
   - Every image band is a **TRANSPARENT cutout by DEFAULT** — sliced from `transparent.png`, `bg` = base hex. In a dark inbox Klaviyo darkens the base and the cutout rides on top, so the design adapts. This is the default for a light/cream + dark-text design; do not bake it flat.
   - Add **`"fill": true`** ONLY when the band is **FULL-BLEED own-background** — it reaches BOTH side edges with its own opaque background and NO base tint shows in its margins (a full-width photo/scene/gradient band, or the dark-logo masthead exception) — OR a transparent cutout of it comes out visibly broken. A filled block is sliced from `color.png` and flattened opaque.
   - **A band that merely CONTAINS a photo but sits on the base with base-tint margins — a testimonial/review card, a photo pair, a badge, a chip — is a CUTOUT, not `fill`.** The photo stays opaque INSIDE the cutout; only the base-tint margins re-theme. Baking such a band from the color export is the rulebook's named COMMON MISTAKE (`slicing_rules.md`). Decide by the MARGINS, never by "it has a photo": trust `plan_facts.json` `images[].dark_hint` + `w_frac` (`w_frac`≈1.0 = full-bleed → `fill`; `<~0.9` = base margins → cutout), and when in doubt confirm with `python3 scripts/imaging.py alphamap builds/<brand>/transparent.png` (OPAQUE margins → `fill`; TRANSPARENT margins → cutout — this is authoritative). Do NOT override a `dark_hint:"cutout"` to `fill`.
   - **Baked DARK text on the bare base fades in dark** (dark-on-dark). A standalone unavailable-font DARK heading you cut out (e.g. a section title) will fade — either accept it and flag to the client, or `fill:true` it as a readable light island; do not end up with baked cards next to a fading cut-out heading (backwards). Text on its OWN light card/pill is fine (the island stays light).
   - **Both** exports go in the plan: `"export":"color.png"`, `"export_transparent":"transparent.png"`. Every block still gets `bg` = base hex (`block_background_color`). Live text + native buttons re-theme automatically via `block_bg` — no `fill` needed.
   - **HARD RULE — dynamic text MUST stay live: any text with a MERGE TAG (`plan_facts.json` `texts[].merge_tag:true`), or meant to personalize at send, is a `role:"text"` block, NEVER baked.** A baked merge tag freezes to literal pixels = broken personalization = verification FAILURE.
   Then classify each horizontal band from `plan_facts.json` (do not re-derive from pixels) and write `plan.json` (schema in `references/klaviyo_api.md` and `scripts/build.py` header): band `range:[y0,y1]` (1.5x px, cut in flat base-tint gaps so slices tile flush — snap tall bands with `imaging.py slice --snap`), `fmt` (jpg = photo/gradient, png = flat/text/line-art), plus text/button blocks inline. **Facts → classification:**
     - a `text` with `available:true` AND `over_image:false` AND `merge_tag:false` → **LIVE `text`** (use its exact `font`/`size`/`color`/`align`; Poppins fallback); `merge_tag:true` → LIVE (hard rule above). **Reproduce EVERY inline style — color, bold, italic.** If the fact has a `runs` array, emit each run with its exact style: `<span style="color:#hex">` for `c`, `<b>` for `b:true`, `<i>` for `i:true` (a two-tone/partly-bold body line or heading is fine as editable spans). If the node-level fact has `italic:true` wrap the whole block in `<i>`; `weight`≥600 → `<b>`. Also eyeball the color overview for anything the runs miss. **NEVER drop a design color OR a bold/italic emphasis** (hard rule: reproduce the design exactly).
     - a `text` with `available:false` (unavailable font), or `over_image:true`, or stylized/two-tone/gradient → part of an **IMAGE** slice. Slice an unavailable-font heading as its own image so adjacent live copy stays live.
     - `images[]` regions are IMAGE bands: transparent **cutout** by DEFAULT; `"fill": true` ONLY for a full-bleed own-background band (`dark_hint:"fill"` AND `w_frac`≈1.0), never for a photo/card that has base-tint margins (see the dark-method bullets above — the alphamap is authoritative).
     - `buttons[]` are pill CANDIDATES, not decisions: use a **native BUTTON** ONLY for a real standalone CTA BELOW the header with a solid fill (take its `label`/`fill`/`radius` from the fact, but **read the label COLOR from the design overview** — a stacked-fill label can report the wrong `text_color`). A pill that is decorative (a small badge like "Verified Customer", a coupon chip), or the header CTA, or has a gradient/photo/textured fill → it stays baked inside its IMAGE slice. CTAs are UNLINKED (empty href).
6. **Build (ONE deterministic call)** — `python3 scripts/build.py builds/<brand>/plan.json` runs slice → compress (parallel) → upload → spec → `build_def` → create/patch → render with no model thinking in the loop (~10–15s). It prints the template id + render path, and a **`dark: N transparent cutouts, M filled`** line + any `DARK-CHECK WARNING`. Each image block is a TRANSPARENT cutout (sliced from `export_transparent`, alpha kept) unless it has `"fill": true` (sliced from `export`, flattened opaque onto `bg`). Real store: put `"store":"<slug>"` in the plan (build.py passes `--store` to every upload/create). To update in place, set `plan.target = {"mode":"patch","id":"<id>"}`. If build.py errors on a block, fix the plan and re-run (idempotent).
7. **Verify (mobile first, LEAN — browser ONLY when there is live text).** build.py already wrote `builds/<brand_slug>/render.html` + printed the `opacity … / DARK-CHECK` line. The visual gate scales to the design's non-deterministic surface:
   - **ALWAYS — browser-free mobile preview (the primary visual gate):** `imaging.py stitch builds/<brand>/plan.json --out builds/<brand>/preview390.png --width 390` stacks the compressed FINAL slices in plan order at 390 → **one visual Read of `preview390.png`** (matches the design: bold/two-tone/forced breaks baked correctly, right slice order, no clipping, no wrong crop). For an all-baked / images+buttons design these ARE the exact pixels Klaviyo renders (slices tile flush, `mobile_margin` 0), so no browser is needed. Then `imaging.py edges builds/<brand>/preview390.png --width 390` (overflow + white side-gutter + mid-email seam; a white-base design reads white — judge overflow/seam only).
   - **ONLY IF the plan has a live `role:"text"` block — add ONE browser pass** (Klaviyo renders live text non-deterministically: wrapping, mobile fit, and the "unsupported formatting → Convert-to-HTML" risk — none of which the stitch can show). Serve (`python3 -m http.server <port> --directory builds/<brand_slug>`, background, note PID; `file://` is blocked), `browser_navigate` to `http://localhost:<port>/render.html`, `browser_resize 390`, `browser_take_screenshot fullPage` → **one visual Read at 390** (the live text wraps/fits, renders bold/breaks, stays editable — no clipping/overflow). Then `browser_resize 600` + screenshot + `imaging.py edges <600-shot> --width 600` for overflow/seam only (no 600 visual Read). `browser_close`, then kill the server PID (`kill <pid>` / `taskkill /PID <pid> /F`, not `pkill`). An all-baked/button design SKIPS this whole browser pass.
   - **Dark mode is NOT rendered or verified visually — Klaviyo does it.** You supply the right INPUTS (transparent cutout slices + `block_background_color` = base hex) and Klaviyo darkens the base and re-themes in a dark inbox; there is no dark screenshot to check (recoloring the browser render misrepresents Klaviyo). Just trust `build.py`'s deterministic **`dark: N transparent cutouts, M filled`** line: a transparent cutout carries alpha (re-themes), a filled block is opaque (own-bg photo / weird cutout), every block has `block_background_color` = base hex. Any `DARK-CHECK WARNING` (a filled block with transparency, or a cutout that came out opaque) is the real bug — re-check the plan's `bg`/`fill`, or spot-check with `imaging.py opacity <file>`. (`dark_preview.js` is deprecated; do NOT gate on it.)
   - `edges` reads: a MID-email full-width white band = a flush-tiling seam to fix; top/bottom white = footer/card, expected; a white-base design reads white throughout (judge overflow/seam only). If `edges` flags a seam, fix the plan `range`/`fmt` and re-run build.py. Iterate until both the visual Read and `edges` are clean.
   - **After a clean verify on a real store, learn the binding:** `klaviyo.py --store <slug> learnfigma --figma-file-key <fileKey>` (`--store` before the subcommand).

## Multi-store: push to the right Klaviyo account
One repo serves many client stores; each is a separate Klaviyo account with its own key in the
SAME `.env`, namespaced `KLAVIYO_STORE_<SLUG>_KEY/_PUBLIC/_NAME/_FIGMA`. For a real client run you
MUST pass `--store <slug>` to EVERY `klaviyo.py upload/create/patch`, resolved BEFORE the first
upload (image assets are per-account). Omitting `--store` falls back to the DUMMY `KLAVIYO_API_KEY`
(the template lands in the dummy account — caught at delivery, not a client leak); never rely on
the default for a client.
- **Resolve (autonomous — bind ONE store or STOP; never guess, never silently use the dummy):**
  1. If you know the store slug (the user named the brand, or a prior run learned it), use
     `--store <slug>`. A wrong/unwired slug hard-fails with the known-store list — surface it, do
     NOT fall back to the dummy.
  2. Else `klaviyo.py whichstore --figma-file-key <fileKey> --brand "<brand read from the design>"`
     → on a confident match prints a tab-separated `slug  name  public  maskedkey  tier` line; on
     NO / ambiguous match it exits non-zero. On non-zero: STOP and report.
  3. Matching is EXACT-normalized, NOT fuzzy: a design's FIRST run usually won't auto-resolve (a
     wordmark rarely equals the slug or org name exactly), so it STOPS — pass an explicit `--store`.
     If unsure of the slug, run `klaviyo.py stores` and match the design's brand to a listed slug.
     There is no mid-run human confirmation — resolution binds ONE store or halts.
- **One slug = store + build dir + template name.** For a real-store run the `<brand_slug>` used for
  `builds/<slug>/` and the template name `<slug>_<template>_<lang>` IS the store slug, so every
  artifact and `--manifest builds/<slug>/manifest` share one folder. Manifest lines carry the
  store's public id; reusing one under a different `--store` hard-fails (assets are per-account).
- **Learn after a clean verify** (so later runs of this design resolve deterministically by file key):
  `klaviyo.py --store <slug> learnfigma --figma-file-key <fileKey>` (`--store` comes BEFORE the subcommand).
- **One-time wiring** (owner, when keys change): export the store sheet (store-name + API-key
  columns) to CSV, drop it at the repo root as `stores_import.csv` (git-ignored), run
  `klaviyo.py wire stores_import.csv` — it live-verifies each key via `GET /api/accounts/`, bakes the
  verified public id + Klaviyo org name into `.env`, and prints a MASKED table (eyeball any `CHECK`
  row where the sheet name != the live org name). DELETE the CSV after. Re-wiring preserves each
  store's learned `_FIGMA`. `klaviyo.py stores` lists what is wired (keys masked); the wired `.env`
  is distributed to teammates like the dummy `.env` (never committed).

## Rules (summary — full detail in references/)
- **Backgrounds:** template + content background = `#FFFFFF` ALWAYS (hard-coded in `build_def.py`). The design's tint comes from each block's `block_background_color` + the full-bleed slice margins, never the canvas. Slices tile FLUSH (no gap) or white shows between them; mobile full-bleed via `mobile_margin:0`. Detail: `references/slicing_rules.md`.
- **Live text (Figma-first):** plain copy + simple headers on a solid fill are live text (BELOW the header — the header section itself is image slices, see "Header + CTAs"). Style it as the design needs (color, bold, italic, alignment, a second-color span) **as long as the text block stays visually editable in the Klaviyo editor** — i.e. it must not trip Klaviyo's "unsupported formatting" notice that forces a Convert-to-HTML block. Use the global heading styles for font-SIZE (so mobile scaling works); inline color spans / `<b>` / `<i>` are fine. Slice text only when it's gradient-filled, over/beside an image, or part of a composite. Keep copy live; use the design's EXACT Figma font so the KM sees the same font as in Figma (read it from `get_design_context`, then `klaviyo.py [--store <slug>] fonts --has "<family>"` prints the ready stack `'<exact font>', 'Poppins', Helvetica, Arial, sans-serif`; Poppins is the fallback). Detail: `references/slicing_rules.md`; fonts: `references/klaviyo_api.md`.
- **Header + CTAs:** the **HEADER section is image slices ONLY** — everything above the CTA as one slice (split if too big) + the main CTA as its own separate image slice; no live text and no native button in the header. Every OTHER CTA (below the header): try a native BUTTON first; if the button block can't reproduce the design (gradient/photo/textured background, or details a button can't render), use an image slice instead. CTAs are left UNLINKED (empty href, native or sliced); the client adds the links in Klaviyo. Detail: `references/slicing_rules.md`.
- **Slicing / split rule / alt text / padding / links:** `references/slicing_rules.md`.
- **Compression:** JPEG q82 for photographic/gradient slices; optimized/quantized PNG for flat/text/transparent (quantize keeps alpha). Targets: ≤200KB JPEG, ≤300KB PNG, ≤250KB smooth-alpha cutout; over a target, SPLIT before over-compressing (`references/slicing_rules.md`). Compress before upload (no API compression).
- **Mobile-first:** verify 390 first. Image blocks full-bleed (horizontal padding 0); native buttons `mobile_stretch_content:true`; body text ≥16px on mobile (EXCEPT a live block forcing the design's exact line breaks, which uses the proportional sub-16 `mobile_font_size` — see `references/slicing_rules.md`).
- **Dark optimization:** transparent cutout slices + each block's `block_background_color` set to the design fill hex, so the inbox re-themes it. Full method + the per-section decision: `references/slicing_rules.md`. Export recipe: `references/figma_and_env.md`.
- **Klaviyo API, the SYSTEM_DRAGGABLE schema, the button schema, the block-spec format:** `references/klaviyo_api.md`.

## Scripts (`scripts/`, run with `python3 <script> -h`)
- `figma_rest.py` — **the REST front-half (needs `FIGMA_TOKEN` in `.env`, scope `file_content:read`).** `fetch <fileKey> <nodeId> builds/<brand> --fonts "..."` writes `color.png` + `transparent.png` + `plan_facts.json` in ~30s (reads + both exports + structured planning facts), replacing the MCP reads + isolation export. `export` = the two PNGs only; `facts` = `plan_facts.json` only. Color render is pixel-identical to the MCP isolation export; transparent is the children composited (SLICE + full-frame bg-rect skipped). MCP is the fallback when REST can't render, and is required for translation (Figma writes).
- `build.py` — **the deterministic one-shot builder: `build.py builds/<brand>/plan.json`** runs slice → compress (parallel) → upload → spec → `build_def` → create/patch → render in ONE call, no model thinking in the loop (~10–15s). The `plan.json` (its header documents the schema) is the single analysis pass the agent writes in step 5. `--dry-run` = slice+compress locally only; `--skip-render` = build without the render. This is the normal path for steps 6–8; the individual scripts below are what it drives.
- `imaging.py` — `compress | detect | gap | crop | dims | overview | shots | alphamap | slice | edges | opacity | stitch`. (`alphamap` = dark-opt cutout/bake margin-alpha map; `slice --ranges` = crop all image bands in one call + print `height600`; `compress --flatten <hex>` = composite onto a solid base → guaranteed opaque bake slice; `edges` = scripted overflow / white-gutter / seam check; `opacity <file>` = OPAQUE vs HAS-ALPHA structural dark check; **`stitch plan.json --out preview390.png --width 390`** = stack the FINAL slices into one browser-free mobile preview for the step-7 visual Read.)
- `phase.py` — optional per-phase wall-clock timer for speed benchmarking: `phase.py builds/<brand> "<label>"` stamps a milestone (start → reads-done → export-done → plan-done → verify-done), `phase.py builds/<brand> --report` prints the table. Not needed for a normal build.
- `dark_preview.js` — DEPRECATED as a verify gate (Klaviyo does dark itself; recoloring the browser render gives false signals — see step 7). Retained only as an optional manual diagnostic.
- `klaviyo.py` — `upload | create | patch | render | get | list | checkenv | stores | fonts | wire | whichstore | learnfigma` (reads the key from env or the repo-root `.env`; **`--store <slug>`** selects a wired account — see Multi-store). `fonts` lists the account's uploaded + web fonts; `fonts --has "<family>"` prints the ready `font_family` stack (`'<exact font>', 'Poppins', Helvetica, Arial, sans-serif`) for a design's font.
- `build_def.py` — expands a block-spec JSON into a full SYSTEM_DRAGGABLE create/patch payload.

## Translation (multi-language replicas)
On request, translate an email/flow into one or more languages by cloning the source design in
Figma and swapping only the words (one replica per language, named `<source name> - <Language>`).
**Order: translate FIRST, then slice — and slice ONLY the translated replica, never the English
source** (the English design is just the translation input; do not build or slice an English
template to "plan" slices). If the client hands over an ALREADY-translated design, skip
translation and slice that design directly. Translation is done at the Figma TEXT layer so copy
destined to become an image slice is translated before the pipeline bakes it; the English source
is never mutated. Each translated replica is then a normal input to the pipeline above (one
template per language).
- **Full method + rulebook:** `references/translation.md`. **Prompt + glossary + memory:**
  `references/translation/` (the `translator` subagent runs the prompt, or translate inline).
- **Non-negotiables:** preserve FULL meaning (fit copy to the fixed layout by re-breaking lines
  and resizing, NEVER by cutting meaning); natural/native copy; lock codes/brands/merge-tags;
  apply locale typography (e.g. "5 %"). Place replicas in EMPTY canvas measured from the whole
  PAGE (`absoluteBoundingBox`), beyond the global occupied rect, with an AABB overlap guard —
  never grow an existing section.
