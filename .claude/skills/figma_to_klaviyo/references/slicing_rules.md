# Slicing + block rulebook

Decision rules for turning a Figma email design into Klaviyo blocks. Process the design top to bottom as an ordered list of regions (a region = a section frame or a visually separated band). Classify each region TEXT, IMAGE, or BUTTON. First matching rule wins. Operational check: if a slice would be mostly plain text on a flat fill, it is the wrong block type — make it live text.

## Dark METHOD — simple, and it is NOT optional (Klaviyo does the re-theming; you supply the inputs)
**Klaviyo does dark mode itself.** You do not render or verify dark — you just ship the right slice + a base-hex `block_background_color`, and Klaviyo darkens the base in a dark inbox and re-themes. So ALWAYS export BOTH the color and the transparent version (step 3) and decide per region by LOOKING at the transparent overview:
- **TRANSPARENT cutout = the DEFAULT for every on-base image** (a card, badge, star row, coupon chip, icon, a photo floating on the base). Slice it from `transparent.png` (alpha kept) and set `block_background_color` = base hex. In dark the base darkens and the cutout rides on top → the design ADAPTS. This is what "dark optimization" means; a light/cream + dark-text newsletter is the ideal candidate — do NOT bake it flat.
- **FILLED (`"fill": true`, opaque) is the EXCEPTION**, used only when: (a) the transparent cutout looks **weird / angled / broken** in the transparent overview, or (b) the region is an **own-background photo** (a full-bleed photo/scene with its own opaque bg — nothing to cut). A filled block is sliced from `color.png` and flattened opaque onto `bg`; it stays a fixed island in dark, which is correct for a photo. When unsure, look at the transparent overview: clean cutout → transparent; weird → `fill`.
- **DARK / STRONG-COLOR BASE (e.g. black, deep maroon)** — the design is already dark. Every image block is `"fill": true` (opaque) with `block_bg` = base hex; no transparent export needed.
- **HARD RULE — MERGE TAGS / dynamic text stay LIVE:** any text with a Klaviyo merge tag (`{{ ... }}`) or meant to personalize at send is a LIVE `text` block, never baked — a baked merge tag freezes to literal pixels = broken personalization = a verification FAIL. Live text + native buttons re-theme automatically via `block_bg`; prefer LIVE text for plain, available-font copy you want editable. Always scan node text for `{{` before making a band an image.
`build.py` prints `dark: N transparent cutouts, M filled` + any `DARK-CHECK WARNING` (a filled block with transparency, or a cutout gone opaque) so a mismatch surfaces deterministically. The `(A)`/`(B)` schematic below is the per-region detail.

## Decision schematic (run per region, top → bottom — this is THE decision; sections below are the detail)
Two questions per region: **(A) which block reproduces it**, then **(B) how it behaves in dark**. Every block always gets `block_background_color` = the section's base hex — that block bg is the ONE dark-theming lever for every block type.

**(A) BLOCK TYPE**
- **HEADER section** (top band, down through the main CTA) → **IMAGE SLICES ONLY** (above-CTA slice + a separate CTA slice; no live text, no native button here — see "Header section")
- Plain copy / heading on a solid fill, BELOW the header, reproducible editably → **LIVE TEXT**
- CTA below the header → **NATIVE BUTTON** (fallback: image slice if a button can't reproduce it; either way left unlinked)
- Anything else — photo, graphic, icon, badge, stars, coupon chip, stylized/gradient text, composite → **IMAGE slice**

**(B) DARK TREATMENT** — only IMAGE slices need a choice (live text + native buttons re-theme automatically via `block_bg`). Read the TRANSPARENT export's MARGIN alpha for the region:
- margins TRANSPARENT — the element sits ON the toggleable base → **TRANSPARENT CUTOUT** (cut from the transparent export) + `block_bg` = base hex. The base re-themes in dark, the element floats. **← THIS IS THE DEFAULT.**
- margins OPAQUE — the region carries its OWN full-bleed background (a full-bleed photo/scene/gradient, base baked into the raster) → **BAKE opaque** (cut from the color export). Stays vivid in dark = correct.
- **Fall back from cutout → bake ONLY when the cutout looks worse:** a dark element/text would go dark-on-dark and vanish in dark (e.g. a strikethrough price); the band straddles base+own-bg (see "STRADDLES"); a quantization artifact degrades it.

**Transparent is the default; baking is a deliberate exception with a named reason — never the lazy default.** Dark-opt with a transparent cutout is NOT optional: always attempt it for an on-base image, and only bake when one of the fallback triggers above actually applies.
**COMMON MISTAKE — do not repeat:** cutting an on-base image (a testimonial card, coupon chip, badge, icon/star row) from the COLOR export so its base margins bake opaque → a light band that won't re-theme in dark. On-base images come from the TRANSPARENT export.

## Precondition: "live-text-eligible"
A text element may become a live TEXT block only if ALL hold:
- Its fill is a SOLID color (not a gradient/linear, not an image fill). Gradient-filled text → image.
- No glyph effects that change appearance (drop shadow, stroke, blur).
- It does NOT sit ON (overlap) an image/photo and is NOT laid out BESIDE one (side-by-side inside a card). **Text STACKED BELOW a photo, on the section's own solid fill, IS live-text-eligible** — "below a photo" is not "over/beside" it (see "Cards" below).
- It can be reproduced with text styling that keeps the block editable in the Klaviyo editor (see "Text styling + fonts"). Two-tone is fine for a heading via a color span; body copy stays one color.

If any fail, the element becomes an IMAGE slice. When a stylized header sits directly above its section content, MERGE it into that section's image slice (don't make a tiny separate header image).

## Use a TEXT block when
- A clean heading / title / subtitle on a solid or flat background, BELOW the email-header section (the top header band is image slices — see "Header section"; "heading" here means a section heading lower in the email, not the email header). Read exact font/size/weight/line-height/color/alignment from Figma. Map headings to the GLOBAL heading styles (for font-size, so they scale on mobile).
- Body / paragraph copy on a solid background.
- **Cards: photo on top, copy below (testimonial / review / feature card) — SPLIT, do not bake.** The photo plus any ornaments stacked with it (stars, quote-mark, badge) = the IMAGE block; the quote/review/caption + attribution sitting BELOW it on the solid fill = a LIVE TEXT block. Cut at the clean gap below the lowest graphic (e.g. below the star row). Only copy that sits OVER or BESIDE the photo forces a slice; copy stacked BELOW does not. Live text here is editable (the KM rotates testimonials), lighter, and re-themes in dark. This is the DEFAULT for stacked photo+text cards.
- **Body copy overlapping the hero's bottom — keep it LIVE if the hero backdrop == base.** A letter/body node's y-range often overlaps the full-bleed hero raster while visually sitting below the scene. Sample the hero's LOWER margin hex: if it equals the section base, cut the hero ABOVE the text and keep the text live — the seam is invisible (photo-bottom == base). Only bake the text in if the overlap region is actual scene content. (Recurs constantly on product-email heroes; without this you needlessly bake live-eligible copy.)
- Any text with a Klaviyo merge tag (e.g. `{{ first_name }}`) → force live text if at all possible, so personalization survives. Only slice it if the font/gradient makes live impossible, and flag it. **BUT a KM-instruction placeholder meant to DISPLAY the literal `{{ … }}` syntax (a dynamic-block note the client will wire up) must be an IMAGE slice** — Klaviyo evaluates merge tags in live text, so the syntax renders to empty in the proof (you get a stray `**`/blank instead of the snippet).
- Plain-text "letter" emails (long stacked paragraphs on flat white): left-aligned, horizontal padding ~18.
- Fix obvious source typos when recreating text live (e.g. a stray-keystroke heading) and flag them; don't reproduce a typo.

## Force an IMAGE slice when
- The text fill is a GRADIENT/linear color (any `GRADIENT_*` fill on a text node) — Klaviyo text can't do gradient fills.
- The font is stylized / not renderable, or the text has glyph effects.
- Text is laid out BESIDE or OVER an image inside a card (bounding-box overlap: the text node's x-range intersects an image node's and they share a vertical band, or the text sits inside an image). Klaviyo's stacked block model can't reproduce side-by-side-in-a-card and would reflow badly on mobile. (Text merely STACKED BELOW a photo does NOT count — that's live text + a separate image block; see "Cards". Don't over-read "card" as "one composite image".)
- Product / hero photography (always an image).
- A full-gradient / full-photo region (the section fill itself is a gradient, or a full-bleed photo background), or a section whose text/CTA sit OVER a full-bleed opaque photo: slice the whole region (split if tall), no native buttons inside, skip dark mode.
- Decorative ornaments that bridge two sections: absorb into the adjacent slice or move the cut to clear them.

## Text styling + fonts
- **Editability is the test.** Style live text however the design needs — color, bold, italic, alignment, a second-color span on a heading — **as long as the text block stays visually editable in the Klaviyo editor.** The hard limit is Klaviyo's "unsupported formatting" notice: if a block trips it, Klaviyo forces a Convert-to-HTML block (no visual editing) — that is the only thing to avoid. Confirmed-editable: `<b>`, `<i>`, `<br>`, `<span style="color:#hex">`, and `text-align`. A two-tone heading via a color span is fine and editable — use it rather than slicing a heading just for a second color. (A two-tone BODY paragraph, or a stylized/gradient heading, still becomes an image.)
- **Prefer global styles for base appearance** (font, size, line-height); use inline styling for per-element needs (a second color, emphasis). In particular, drive heading font-SIZE from the global heading styles (which carry `mobile_font_size`) — an inline font-size disables mobile scaling and overflows.
- **Fonts (use the design's exact Figma font; Poppins as fallback):** the KM must see the same font in Klaviyo as in Figma. Read the exact Figma family from `get_design_context`, then `klaviyo.py [--store <slug>] fonts --has "<family>"` prints the ready `font_family` stack — `'<exact font>', 'Poppins', Helvetica, Arial, sans-serif` — to paste into the spec. Both the account's UPLOADED fonts (`@font-face`, any name) AND its WEB fonts (`@import` Google fonts the editor offers, e.g. Poppins) count as the exact font; only if the account has neither does the stack lead with Poppins. Keep copy live either way (never slice over font fidelity). Detail: `references/klaviyo_api.md`.
- **Match every live-text block to Figma EXACTLY, then verify in the render.** Pull the precise values from `get_design_context` + `get_metadata` and reproduce them, don't approximate: font-size, line-height (the `leading-[…]` value), the design's LINE BREAKS (force them with `<br>` — don't let copy free-flow when the design sets specific breaks), two-tone color spans, bold/italic. Then size the block to the design's measured HEIGHT: a fixed-height bar/frame (e.g. a 43px trust bar = text line-box + top/bottom padding → set line-height + padding to hit it; a multi-line block = lines × line-height + paragraph gaps + padding). Open the proof and confirm everything matches: bold/italic actually render, breaks land where the design's do, one-line copy stays one line (drop `letter_spacing`, trim padding, set a small `mobile_font_size`), and the block/bar height equals the Figma height. A wrap the design didn't have, emphasis that didn't render, or a wrong block height is a fidelity bug — catch it here.

## Header section (top band — image slices only; simplest rule)
The HEADER is the top band of the email, from the top edge down to and INCLUDING the main header CTA. Handle it as IMAGE SLICES ONLY — never live text, never a native button in this region:
- **Everything ABOVE the CTA** (logo / masthead + headline + any hero art) = ONE image slice. If that slice exceeds the size cap in "The split rule" (too tall / too heavy), split it into 2-3 slices at the flattest legal gaps.
- **The main CTA** = its OWN separate image slice (full-bleed, tiling flush with its neighbors), left UNLINKED — the client adds the link in Klaviyo (an unlinked CTA slice renders as a plain image, which is intended). The label bakes in (the KM changes the link, not the wording, without a re-slice).
- So a header is normally exactly two slices: `[above-CTA]` + `[CTA]` (more only if the above-CTA part is too big and must split).

This OVERRIDES the live-text-eligibility and native-button rules for the header region ONLY. Below the header CTA, the normal per-region rules resume (live text where eligible, native buttons for other CTAs). Dark treatment of each header slice follows the usual margin-alpha rule (`block_background_color` = base hex; cutout vs bake per its margins; a dark logo on the bare base bakes as a light masthead — see the dark-opt EXCEPTION).

## CTAs / buttons (decide per CTA, BELOW the header)
- **The header CTA is its own separate image slice (unlinked), and the whole header is image slices** — see "Header section" above. Everything below covers the OTHER CTAs (mid-email, repeat-offer, footer).
- **Try a native BUTTON first** (for non-header CTAs). Use a native button whenever the button block can faithfully reproduce the CTA — a solid-fill pill + text + radius, on a background you can match with `block_background_color`. A native button is editable (label/link/color/size), crisp at any density, a real tap target, and accessible. Set `block_background_color` to the surrounding base hex so it sits seamlessly between neighbors.
- **Slice it as an image instead when the button block can't reproduce the design:** a gradient or photo background behind/around the pill, a textured/gradient pill, or any detail a button block can't render. Then divide the section at the button (upper-content slice + button slice).
- **CTAs are left UNLINKED** — the href stays empty on native buttons AND on image-slice CTAs alike; the client adds the real links in Klaviyo. (Classification — native button vs image slice — is unchanged; only the link VALUE is empty. A native button with no link still renders as a visible pill; an unlinked image-slice CTA renders as a plain image.)
- **Native button fields** (build_def `button` block): `label`, `fill` (pill color), `color` (text), `block_bg` (surrounding base hex), `radius`, `font_family`, `font_size`, `weight`, `letter_spacing` (INTEGER), `pad` `[vertical,horizontal]`, `mobile_stretch_content:true`. The `href` is left empty (omit it). Renders content-width + centered on desktop and mobile.
- Detection note: `imaging.py detect` is polarity-agnostic (finds a pill lighter OR darker than its bg; label text doesn't split it) but returns false positives on busy/photographic slices — the real button is the band whose `pill` color is clearly distinct from the surrounding bg and which has a distinct `text` color. A pill with dark label text shows as TWO bright bands (the text interrupts the run); treat them as one pill.

## The split rule (do not keep big composites as one slice)
Big composite images bloat the email and read as spam. Split, but byte/height-driven, not literally "half":
- **Slice at every OBVIOUS internal background gap** (between steps, list items, sub-sections), even under the size thresholds. A tall multi-part section becomes 2-3 slices cut at the between-item gaps. Keep a region whole only when it has no clean internal gap.
- Size cap that FORCES a split: a slice over ~1200 CSS px tall at 600 width, or over ~200KB (JPEG) / ~300KB (PNG). Recurse on the pieces.
- **Tall image cards → 2-3 slices:** even under 1200px, a single card that reads as "really big" (>~700 CSS px / >1050px at 1.5x) should split for faster load + lower spam risk. Cut at the flattest row (a stdev-per-row scan over the middle band finds it; a flat gap between sub-elements is ideal) even when it is not a full-width background gap. Adjacent crops of the SAME image reassemble seamlessly (sliceA.end == sliceB.start), so the split is invisible.
- Cut lines MUST fall in a background gap (a near-uniform full-width band with no text/photo/button/ornament crossing that y). Use `imaging.py gap` and cross-check against the Figma node positions (node y-values can have negative offsets/overflow, so don't trust geometry alone). Once the image-band boundaries are set, cut them ALL in one pass with `imaging.py slice <export.png> <outdir> --ranges y0:y1,y0:y1,... [--names ...] [--snap]` (list only the image bands; live-text gaps are omitted; `--snap` snaps each interior boundary to the flattest nearby row; it prints each block's `height600` for the spec).
- NEVER cut through a face, product, logotype, button, or ornament. If no legal gap exists, don't split; compress harder.
- Don't over-slice: each block adds HTML. Keep generated HTML well under 85KB (Klaviyo risk) / 102KB (Gmail clips). Prefer merging adjacent slices over shattering.

## Backgrounds
- **Template + content background = `#FFFFFF` ALWAYS** — hard-coded in `build_def.py` (the spec's `background_color`/`content_background_color` are ignored on purpose). Never a non-white canvas, for any design.
- **The design's real base color is painted ON TOP of the white canvas** by (a) each block's `block_background_color` and (b) the full-bleed image slices' own margins — never the canvas. Sample the base hex (edge-color scan down the design) and set it as every block's `block_background_color`; full-bleed slices already carry it. With flush blocks (no inter-block gaps) the white canvas never shows.
- **Slices must TILE flush.** Any vertical gap between two image slices exposes a white stripe (the canvas). Make consecutive slices touch (sliceA.end == sliceB.start); live text blocks (bg = base tint) fill the spans between slices.
- **Full-bleed on mobile — `mobile_margin:0`** (build_def default). Klaviyo's default `mobile_margin:10` leaves a 10px white canvas gutter down both sides on mobile (invisible on a white design, visible as white side-strips on any colored design). Verify on a real ~390px emulation by scanning the LEFT/RIGHT edge pixels (x≈4) for white.
- For a block over a colored design, set `block_background_color` to the region's color (sample the exact hex; for a gradient, region-match each block to its vertical position so seams vanish). For a native button between two image slices, place the cut lines in clean background BEYOND the pill's shadow/glow so neither edge carries it, and set `block_background_color` to the average of the two clean edge rows (the color before + after the block) — never a transient shadow/glow.
- **RED FLAG — a `#444444` `block_background_color` (or slice margin) is the Figma-page gray leaking, never a real design color.** It comes from a `download_assets defaultScale` export that baked the gray page into empty areas. Resample the real fill from the ISOLATED export / design pixels. Quick check before shipping: grep the payload for `#444`.

## Alt text
- Use the slice region's visible text (concatenate the text nodes in reading order; trim ~120 chars; strip merge tags). A pure photo/decorative slice → `"Loading..."`.

## Links
- **All CTAs are left UNLINKED** — the href is empty on native buttons and on CTA image slices (a product card's SHOP NOW, a bordered CTA card) alike; the client wires the real links in Klaviyo. Pure image slices with no CTA, and live text blocks, also carry no link. Still CLASSIFY CTAs as before (native button vs image slice) — only the link value is empty.

## Padding
- Read exact padding from Figma geometry: left/right inset = distance from content edges; top/bottom = background gap to neighbors. Defaults (fallback only): horizontal 0 for full-bleed blocks; ~18 for plain-text letter bodies; image blocks horizontal padding 0 (full-bleed).
- **Text block has TWO paddings (do not confuse them):** the spec `pad` feeds the **BLOCK padding** (`block_padding_*` — the outer cell that carries the block bg, where the Figma layout spacing lives); the **TEXT-AREA padding** (`inner_padding_*` — the inner text cell) is always forced to **0/0** by `build_def.py` (not read from the design) so copy sits flush inside the block. (The button block's `inner_padding_*` is its own pill padding — a separate thing, unaffected.)

## Mobile-first
- Verify 390 first; desktop (600) is the secondary gate.
- Headings: global heading styles carry desktop `font_size` + `mobile_font_size`; don't set inline font-size (it disables mobile scaling).
- Buttons: `mobile_stretch_content:true`. Body text ≥16px on mobile BY DEFAULT — but when a live-text block forces the design's exact line breaks (`<br>`), set `mobile_font_size` to the design's PROPORTIONAL size instead (≈ `desktop_size × 390 / designWidth`, ~0.65×, e.g. 20px→13px), and scale the side padding the same way, so each forced line still fits ONE line at ~390 and doesn't re-wrap into orphan words. The mobile-design is the fixed-width design scaled to the phone, so this proportional size can dip below 16px and that's correct. Verify in the render that NO design line wraps (watch out: `height ÷ line-height` over-counts because the `<p>` paragraph margin is included — confirm by eye / per-line rects, not just the ratio).

## Dark optimization (transparent-cutout method)
Klaviyo SYSTEM_DRAGGABLE emits NO dark-mode CSS. The adaptation lever is the block background, not head CSS. Apple Mail respects dark and does not invert images; Gmail ignores it.

**Mechanism.** Ship TRANSPARENT cutout slices AND set every block's `block_background_color` to the design's exact fill hex. That block bg is a real CSS color the inbox's dark/light engine re-themes; the transparent cutout (and any live text) rides on top. Light = looks like the design; dark = the inbox darkens the block bg and the cutouts + live text re-theme with it. Plain transparent on the white canvas is NOT enough (copy drops to white) — the themeable `block_background_color` is what makes it work. A slice you WANT to re-theme must be transparent — an opaque slice stays a fixed-color island in dark mode. `build_def.py` sets `block_background_color` on image blocks too (from the spec `bg` field).
- **EXCEPTION — a DARK logo/wordmark on the BARE base must be BAKED, not a cutout.** A dark raster can't re-color, so as a transparent cutout it goes dark-on-dark and vanishes in dark mode (same failure class as "baked dark text floating on the base" below). Bake it (with the title) as ONE light *masthead* slice so it stays visible in BOTH modes. Tell which case you're in from the header region's margin alpha: header OVER the hero photo (opaque margins) → it bakes into the hero anyway; header on the bare base (transparent margins) → bake a separate masthead. (A light/white logo, or one that bakes into a photo, is fine. Don't reflexively cut a dark logo out.)

**Candidates.** Light designs (cream/white/pale fill + dark text/photos) adapt via cutouts. A full-gradient/full-photo region, or a section whose text/CTA sit over a full-bleed opaque photo, can't be a cutout → slice it opaque and skip dark mode for it. Not every section adapts, and that is fine (a photo banner staying vivid in dark is correct).

**Per-section decision = read the TRANSPARENT export's MARGIN alpha, top-to-bottom (the margin-alpha map IS the analysis — don't judge by compositing on a theme).** Get the map deterministically with `imaging.py alphamap <transparent.png>` (per-band mean margin alpha → `TRANSPARENT` = on-base cutout / `OPAQUE` = own-bg bake / `MIXED` = a boundary/straddle you decide). It only MEASURES; the bake/cutout call on a `MIXED` band stays your judgment. The two base cases (transparent margins → cutout = DEFAULT; opaque margins → bake) are in the Decision schematic (B) at the top. The edge cases below refine it:
- **STRADDLES the boundary** (a colored band baked behind only PART of the slice — e.g. a tinted rect behind the top of a Facebook-post card — while the rest of the margin is transparent) → emit ONE COLOR-FILLED ("back-filled") opaque slice from the color export. Else the transparent half re-themes and the baked half doesn't = a split block. Check the WHOLE slice's margin top-to-bottom, not one row; when in doubt on a card over any tinted band/shadow, prefer the back-filled slice.
- **Toggle ONLY the page background; bake every design element.** In the isolation export turn off ONLY the page-background fill and leave all design fills ON (cards, panels, bands, shapes, pills), so each bakes into its cutout on the page-base `block_background_color` (which re-themes). This is why an inset/rounded/arched card stays intact — it bakes as a cutout on the re-theming base; its own color stays baked (correct: e.g. a green card pops on a darkened page). NEVER turn a card's fill off to flatten it into a block bg — a rectangular block bg can't reproduce an inset or rounded card, so the shape vanishes. Detail: `figma_and_env.md`.
- **Soft glow / drop shadow / halo on a LIGHT base = a DARK-MODE-NATIVE effect → TRANSPARENT cutout, NEVER bake (e.g. a product floating on the page base).** A light-on-light glow (a white ellipse behind a product, a soft drop shadow, a feathered photo edge) is INVISIBLE in light mode, so a baked-opaque version looks identical in the light render — it gives ZERO warning anything is lost. But that glow only appears once the background darkens: it IS the dark-mode effect, and the transparent-cutout method exists to deliver exactly it. Baking the section on the light base silently kills the glow AND leaves a fixed light island in dark. Ship it as a transparent cutout (`block_background_color` = base hex): identical in light, glow blooms against the darkened bg. **Tell from the margin-alpha map:** a broad band of SOFT / PARTIAL alpha (an anti-aliased gradient, not a hard 0%↔100% edge) around a photographic element IS a glow/shadow — preserve it via the cutout; never read soft alpha as "a halo/blob to avoid by baking" (that inverts the method — one floating-product hero had a ~58%-partial-alpha white-ellipse glow that was measured and still baked, the canonical mistake). This OVERRIDES the "product/hero photography → image" and "photo stays as-is in dark is fine" heuristics: those hold only for a photo with its OWN opaque background (hard ~100% margins); a product FLOATING on the page base (transparent or soft-alpha margins) is a CUTOUT. The margin-alpha map decides, not the word "photo." Compression: a photographic cutout with a smooth-alpha glow CANNOT be palette-PNG'd (256 colors band the alpha into concentric RINGS) — `imaging.py compress` now auto-detects >8% partial-alpha and switches to a truecolor path (quantize RGB only, keep 8-bit alpha, downscale to a byte cap), so just compress as usual.

**TEXT vs IMAGE inside a section is a FIDELITY call, not a contrast judgment** (`block_background_color` handles theming either way): plain text on a solid fill → TEXT block; gradient/droplets/icons/badges/photos → IMAGE block (transparent cutout).

**A transparent image cut MUST clear a heading's FULL vertical extent.** A centered heading leaves transparent side-margins, so a naive "most-transparent-row" cut lands inside the heading band, baking its bottom into the slice AND duplicating the live text. Cut below the heading's bottom edge (cross-check the Figma node's y-extent, not just the alpha map).

**Compression of cutouts.** A transparent photo cutout (RGBA PNG) is heavy; `imaging.py compress` auto-quantizes RGBA to a 256-color palette (FASTOCTREE, keeps alpha) — a big size win with no visible loss at email scale. Verify a margin pixel stays `alpha==0`. (JPEG is not an option for a cutout — no alpha.)

**Display-font designs: STILL apply the per-section cutout method; know where baked text fades.** Even when headings use a DISPLAY font that must be baked (e.g. Bebas Neue — condensed all-caps, not hosted), do NOT default the whole email to opaque (that throws away the dark re-theming the client set the transparent version up for). Apply the normal per-section decision from the transparent export's margin alpha: page-base sections → TRANSPARENT cutout + `block_background_color`; own-fill sections (full-width photo, a colored panel like a hero band) → bake. The real limitation is narrower than "no dark-opt": a baked DARK-text element FLOATING on the page base (a standalone heading; the copy inside a text-beside-photo composite) goes dark-on-dark in dark mode, because a baked font can't re-color (only LIVE text does). But baked text sitting on its OWN light card / panel / pill is FINE — that island stays light in dark and the text stays readable (the card even pops on the darkened page; verified on a bonus-card + checklist-pills layout). So: cut so text rides on baked islands where the design provides them (cards, pills); for standalone display headings on the bare base, either accept the dark-mode fade or make them LIVE web-safe text (font-fidelity trade-off). Keep genuinely-plain paragraphs live (they re-theme correctly). Flag the heading trade-off to the client — don't silently pick opaque-everything OR fade-prone cutouts.

**Hard rules.**
- Get both exports via the ISOLATION recipe (`references/figma_and_env.md`). NEVER `download_assets defaultScale`/`defaultFormat` for a light design (it bakes the gray Figma page into empty areas) and it can't produce a transparent export at all.
- NEVER fake transparency with local tools (manual color removal). It must come from the Figma fill-off isolated render. If the file is VIEW-ONLY (`use_figma` writes fail), FLAG it and stop; don't substitute a local strip.
- Keep the slice-length rule: ≤~700 CSS px (≤1050px at 1.5x); hard cap 1200. Tall sections still split at the flattest legal gap.

**Mixed light/dark designs** (e.g. a colored hero + white cards): the white canvas is still correct (blocks supply all color); apply the per-section decision band by band — set each block's `block_background_color` to its fill hex and ship its cutout transparent, color-filled only for a straddling band.

**Verify (step 9).** Render light, then `browser_evaluate` a DARK preview: recolor every `block_background_color` to a darkened version of the fill and (for a light design with dark text) lighten the dark live text; live text + cutouts must re-theme uniformly, no marooned near-white box. Known Klaviyo quirk: a transparent PNG can render inside a white container box in iOS Mail dark mode — if so, set the injected block background to none/transparent or fall back to an opaque slice.
